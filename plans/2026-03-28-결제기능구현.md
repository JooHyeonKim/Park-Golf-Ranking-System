# 결제 기능 구현 정리

## 개요

파크골프 스코어 집계 프로그램에 월 3,000원 구독 결제 기능을 추가했다.
회원가입 후 30일 무료 체험 → 이후 결제 필수. 결제 게이트웨이는 PortOne(포트원, 구 아임포트)을 사용한다.

---

## 1. PortOne 설정

### 1-1. 가맹점 등록
- [admin.portone.io](https://admin.portone.io) 에서 회원가입
- 사업자등록 전에도 **테스트 모드**로 개발 가능
- 실 결제를 위해서는 사업자등록증 + 통신판매업 신고증 필요

### 1-2. 가맹점 식별코드 확인
- 관리자 콘솔 → **결제 연동** → **내 식별코드·API Keys**
- `imp_XXXXXXXX` 형식의 가맹점 식별코드
- REST API Key / REST API Secret 도 같은 페이지에서 확인

### 1-3. PG사 채널 연동
- 관리자 콘솔 → **결제 연동** → **테스트 연동 관리** → **PG사 추가**
- **카카오페이** 선택 (정기결제 지원)
- PG 상점아이디: 비워두기 (테스트 모드에서 자동 할당)
- 수동 승인: 사용 안 함

### 1-4. 키 값 정리

| 키 | 용도 | 저장 위치 |
|----|------|----------|
| 가맹점 식별코드 (`imp_...`) | 클라이언트에서 SDK 초기화 | `.env` → `VITE_PORTONE_MERCHANT_ID` |
| REST API Key | Edge Function에서 결제 검증 | Supabase Secrets → `PORTONE_API_KEY` |
| REST API Secret | Edge Function에서 결제 검증 | Supabase Secrets → `PORTONE_API_SECRET` |

> **주의**: REST API Key/Secret은 절대 클라이언트(.env의 VITE_ 변수)에 넣으면 안 됨. 서버(Edge Function)에서만 사용.

## 2. Supabase 설정

### 2-1. 테이블

#### subscriptions
```sql
CREATE TABLE subscriptions (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE UNIQUE,
  status TEXT NOT NULL DEFAULT 'trialing'
    CHECK (status IN ('trialing', 'active', 'past_due', 'canceled', 'expired')),
  trial_ends_at TIMESTAMPTZ NOT NULL,
  billing_key TEXT,
  current_period_start TIMESTAMPTZ,
  current_period_end TIMESTAMPTZ,
  canceled_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

#### payment_history
```sql
CREATE TABLE payment_history (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  amount INTEGER NOT NULL,
  portone_imp_uid TEXT,
  merchant_uid TEXT,
  status TEXT NOT NULL CHECK (status IN ('paid', 'failed', 'refunded')),
  paid_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW()
);
```

### 2-2. RLS 정책
- 유저는 자신의 subscription/payment_history만 SELECT 가능
- INSERT/UPDATE는 Edge Function(service_role)만 가능

```sql
ALTER TABLE subscriptions ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users can read own subscription"
  ON subscriptions FOR SELECT USING (auth.uid() = user_id);

ALTER TABLE payment_history ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users can read own payment history"
  ON payment_history FOR SELECT USING (auth.uid() = user_id);
```

### 2-3. 자동 구독 생성 트리거
회원가입 시 subscriptions 행을 자동 생성 (30일 무료체험):

```sql
CREATE OR REPLACE FUNCTION handle_new_user_subscription()
RETURNS TRIGGER AS $$
BEGIN
  INSERT INTO public.subscriptions (user_id, status, trial_ends_at)
  VALUES (NEW.id, 'trialing', NOW() + INTERVAL '30 days');
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;

CREATE TRIGGER on_auth_user_created_subscription
  AFTER INSERT ON auth.users
  FOR EACH ROW EXECUTE FUNCTION handle_new_user_subscription();
```

### 2-4. updated_at 자동 갱신 트리거
```sql
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER update_subscriptions_updated_at
  BEFORE UPDATE ON subscriptions
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
```

### 2-5. pg_cron (만료 자동 처리)
매일 자동으로 만료된 체험/해지 구독을 expired로 전환:

```sql
CREATE EXTENSION IF NOT EXISTS pg_cron;
CREATE EXTENSION IF NOT EXISTS pg_net;

SELECT cron.schedule(
  'expire-subscriptions',
  '0 0 * * *',
  $$
    UPDATE public.subscriptions
    SET status = 'expired', updated_at = NOW()
    WHERE status = 'trialing' AND trial_ends_at <= NOW();

    UPDATE public.subscriptions
    SET status = 'expired', updated_at = NOW()
    WHERE status = 'canceled' AND current_period_end <= NOW();
  $$
);
```

### 2-6. Supabase Secrets 등록
Supabase Dashboard → **Edge Functions** → **Secrets** 탭에서 추가:

| Name | Value |
|------|-------|
| `PORTONE_API_KEY` | PortOne REST API Key |
| `PORTONE_API_SECRET` | PortOne REST API Secret |

> `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`는 자동 제공됨.

### 2-7. Authentication URL 설정
로컬 테스트 시 OAuth 로그인을 위해:
- Supabase Dashboard → **Authentication** → **URL Configuration** → **Redirect URLs**
- `http://localhost:5174/**` 추가

## 3. Edge Functions

3개의 Edge Function을 Supabase에 배포. 모두 `verify_jwt: false`로 설정 (함수 내부에서 직접 인증 처리).

### 3-1. verify-payment
- 경로: `supabase/functions/verify-payment/index.ts`
- 역할: 클라이언트에서 결제 완료 후 호출. PortOne API로 결제를 검증하고 구독을 활성화
- 흐름:
  1. Authorization 헤더에서 유저 토큰 추출 → user_id 확인
  2. PortOne API 토큰 발급 (`/users/getToken`)
  3. 결제 정보 조회 (`/payments/{imp_uid}`)
  4. 금액 3,000원 + 상태 paid 검증
  5. subscriptions 테이블 업데이트 (status=active, billing_key 저장, period 설정)
  6. payment_history에 기록

### 3-2. cancel-subscription
- 경로: `supabase/functions/cancel-subscription/index.ts`
- 역할: 유저가 구독 해지 시 호출
- 흐름:
  1. 유저 인증 확인
  2. 현재 구독 상태가 active인지 확인
  3. status를 canceled로 변경 (current_period_end까지 이용 가능)
  4. PortOne 빌링키 삭제 (선택적)

### 3-3. recurring-billing
- 경로: `supabase/functions/recurring-billing/index.ts`
- 역할: pg_cron에서 매일 호출. 만료된 활성 구독에 대해 빌링키로 재결제
- 흐름:
  1. service_role 키로 인증 확인 (외부 호출 차단)
  2. current_period_end가 지난 active 구독 조회
  3. 각 구독에 대해 PortOne `/subscribe/payments/again` API 호출
  4. 성공: period 갱신, payment_history 기록
  5. 실패: past_due → 재시도 후 expired 전환
  6. 무료체험/해지 만료 유저도 expired로 전환

### Edge Function 배포 방법
Supabase CLI 또는 MCP 도구로 배포:
```bash
supabase functions deploy verify-payment
supabase functions deploy cancel-subscription
supabase functions deploy recurring-billing
```

### 4-3. 앱 게이트 흐름

```
로딩 → 미인증 → 로그인 화면
        인증 → 마이그레이션 → 구독 확인
                               ├─ trialing → 메인 앱 + 잔여일 배너
                               ├─ active → 메인 앱
                               ├─ canceled (기간 남음) → 메인 앱
                               └─ expired → 결제 필수 화면 (차단)
```

### 4-4. 결제 흐름 (클라이언트)

1. `index.html`에서 PortOne SDK 로드 (`https://cdn.iamport.kr/v1/iamport.js`)
2. `main.jsx`에서 `initPortOne()` 호출 → `IMP.init(merchantId)`
3. 유저가 "구독하기" 버튼 클릭
4. `portone.js`의 `requestSubscriptionPayment()` 호출
5. `IMP.request_pay()` → 카카오페이 결제 팝업
6. 성공 콜백에서 `imp_uid`를 받아 `verify-payment` Edge Function 호출
7. Edge Function이 결제 검증 후 구독 활성화
8. 클라이언트에서 `refetch()` 호출 → 구독 상태 갱신 → 메인 앱 표시

## 5. 구독 상태 흐름

```
회원가입 → trialing (30일)
              │
              ├─ 30일 경과 (미결제) → expired (차단)
              │
              └─ 결제 완료 → active
                               │
                               ├─ 30일 후 자동 재결제 → active (갱신)
                               │
                               ├─ 재결제 실패 → past_due → expired
                               │
                               └─ 유저가 해지 → canceled
                                                  │
                                                  └─ period 종료 → expired
```

## 8. 테스트 방법

1. **무료체험 확인**: 새 계정 가입 → subscription 자동 생성 → 30일 내 앱 정상 이용
2. **만료 시뮬레이션**: Supabase에서 `trial_ends_at`을 과거로 변경 → 차단 화면 표시
3. **결제 테스트**: PortOne 테스트 모드 → 카카오페이 결제 → 구독 active 전환
4. **해지 테스트**: 구독 관리 → 해지 → period 종료 후 차단
5. **재구독 테스트**: 해지 후 다시 구독하기 → 새 빌링키 발급 → 구독 재활성화

## 9. 트러블슈팅

### Edge Function 401 에러
- 원인: Supabase의 JWT 검증(`verify_jwt: true`)이 요청 차단
- 해결: Edge Function을 `verify_jwt: false`로 배포 (함수 내부에서 직접 인증 처리)

### "이미 결제가 이루어진 거래건입니다" 에러
- 원인: `customer_uid`를 userId 고정값으로 사용하여 빌링키 충돌
- 해결: `customer_uid`에 타임스탬프 추가 (`cuid_${userId}_${Date.now()}`)

### 결제 검증 실패
- 원인: Edge Function 호출 시 `apikey` 헤더 누락
- 해결: fetch 호출에 `apikey: SUPABASE_ANON_KEY` 헤더 추가
